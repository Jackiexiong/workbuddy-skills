# 简历筛选评分标准指南

本文档供 AI 在评分简历时参考，确保评分的一致性和可解释性。

---

## 总体评分公式

```
总分 = 硬性条件(达标/淘汰) 
     + 技能栈匹配 × 25%
     + 项目/经验匹配 × 20%
     + 软性素质 × 10%
     + 行业/公司背景 × 10%
     + 简历质量 × 5%
```

- **硬性条件为"淘汰"**：总分计为 0，标记为"不匹配"，输出淘汰原因
- **硬性条件为"通过"**：其他 5 项按权重加权

---

## 一、硬性条件匹配（权重 30% / 一票否决）

根据 JD 中的硬性门槛判定"通过"或"淘汰"。

### 常见的硬性门槛类型

| 门槛类型 | 举例 | 判定标准 |
|----------|------|----------|
| 学历要求 | "本科及以上" | 本科/硕士/博士 → 通过；大专 → 淘汰 |
| 专业要求 | "计算机相关专业" | 计算机/软件/电子/通信/数学等 → 通过；非相关 → 淘汰或标记减分 |
| 工作年限 | "3 年以上" | ≥3 年 → 通过；<3 年 → 淘汰或减分 |
| 特定证书 | "持有 CPA 证书" | 有 → 通过；无 → 淘汰 |
| 语言要求 | "英语六级以上" | CET-6 / 雅思 / 托福 通过 → 通过 |

### 边缘情况处理

- **"本科及以上 + 经验丰富者大专也可"**：大专但有突出项目经验 → 标记为"破格候选"，其他维度评分后由用户决定
- **年限仅差少量**（要求 5 年，候选人有 4 年）：减分处理（扣除硬性条件权重的 30%），不直接淘汰
- **JD 未明确写学历要求**：将此维度标记为"未明确"，不淘汰。根据行业默认标准（如技术岗默认本科）标注"建议需确认"

---

## 二、技能栈匹配（权重 25%）

### 评分标准（1-10 分）

| 分数 | 描述 |
|------|------|
| 9-10 | JD 要求的核心技能全部精通，且有额外加分技能 |
| 7-8 | JD 要求的核心技能大部分掌握，少数需加强 |
| 5-6 | JD 要求的核心技能掌握一半左右 |
| 3-4 | 仅掌握 JD 要求的少量技能 |
| 1-2 | 技能栈与 JD 要求严重不匹配 |

### 打分要点

1. **区分"精通"与"了解"**：简历中写"精通" ≠ 真精通，但评分时按简历自述给分
2. **核心技能权重更高**：JD 中反复提及或排在首位的技能权重更高
3. **技能迁移能力**：如 JD 要 Java，候选人是 C# → 可迁移，酌情给分（不精通但相关）
4. **版本/框架匹配度**：JD 要 Spring Boot 3.x，候选人是 Spring Boot 1.x → 匹配但需提醒版本差距

---

## 三、项目/经验匹配（权重 20%）

### 评分标准（1-10 分）

| 分数 | 描述 |
|------|------|
| 9-10 | 项目经验与 JD 职责高度吻合，且项目规模/复杂度突出 |
| 7-8 | 项目经验与 JD 职责大部分吻合 |
| 5-6 | 部分项目经验相关，但有明显领域差异 |
| 3-4 | 项目经验关联度低 |
| 1-2 | 无相关项目经验 |

### 打分要点

1. **项目规模**：高并发/大数据量/大规模用户 → 加分
2. **项目角色**：技术负责人/架构师 → 加分；参与/辅助 → 正常评分
3. **项目产出**：有可量化的产出指标（性能提升 50%、日活增长等）→ 加分
4. **完整度**：参与过完整项目生命周期 → 加分
5. **行业匹配**：JD 是互联网金融，候选人有金融项目经验 → 高分

---

## 四、软性素质（权重 10%）

### 评分标准（1-10 分）

| 分数 | 描述 |
|------|------|
| 9-10 | 简历体现优秀的沟通/协作/领导力，有管理经验或团队贡献 |
| 7-8 | 有明确的团队协作和跨部门合作经历 |
| 5-6 | 有基本的团队合作描述 |
| 3-4 | 仅描述个人工作，无团队相关表述 |
| 1-2 | 无软性素质相关任何信息 |

### 打分要点

1. **管理经验**：带过团队、项目负责人 → 加分
2. **沟通能力**：有客户对接、技术分享、培训经历 → 加分
3. **学习能力**：转行成功、快速上手新技术 → 加分
4. **主动性**：有开源贡献、技术博客、专利 → 加分

---

## 五、行业/公司背景（权重 10%）

### 评分标准（1-10 分）

| 分数 | 描述 |
|------|------|
| 9-10 | 同行业头部公司同岗位经验 |
| 7-8 | 同行业公司经验或非同行业头部公司 |
| 5-6 | 同行业相关经验 |
| 3-4 | 不同行业但有可迁移经验 |
| 1-2 | 跨行业且无关联 |

### 加分项

- BAT/TMD/字节/华为等大厂背景 → +2
- 知名外企 → +1
- 创业公司核心成员 → +1
- 与 JD 完全相同的行业赛道 → +2

---

## 六、简历质量（权重 5%）

### 评分标准（1-10 分）

| 分数 | 描述 |
|------|------|
| 9-10 | 结构清晰、信息完整、量化产出多、排版专业 |
| 7-8 | 结构完整、信息较全、有量化数据 |
| 5-6 | 基本信息完整但缺乏量化 |
| 3-4 | 信息不完整、结构混乱 |
| 1-2 | 严重缺项、无法判断候选人基本情况 |

### 打分要点

1. **结构完整性**：个人信息、教育背景、工作经历、项目经历、技能清单 → 是否齐全
2. **量化程度**：是否有数据支撑（"提升性能 50%" vs "提升了性能"）
3. **排版专业度**：整洁、无错别字、格式统一
4. **简历针对性**：是否针对应聘岗位做了针对性调整

## 特殊标记规则

在评分过程中，遇到以下情况应做特殊标记：

| 标记 | 含义 | 处理方式 |
|------|------|----------|
| ⭐ **破格候选** | 硬性条件差但其他维度突出 | 在报告中标注"建议破格考虑"，单独列出 |
| ⚡ **快速响应** | 候选人目前处于离职状态，可快速入职 | 在报告中标明"可快速到岗" |
| 🔄 **转行候选** | 从其他领域转行，但有强学习能力证明 | 标注转行背景，由用户判断 |
| ⚠️ **信息存疑** | 简历信息有疑点（日期矛盾、过度夸大） | 标注疑点，建议面试时重点考察 |
| 📋 **重复简历** | 相同候选人多渠道投递 | 去重处理，只保留一份 |

## 评分示例

### 示例：高级 Java 后端工程师

**JD 关键要求：**
- 本科及以上，计算机相关专业
- 5 年以上 Java 开发经验
- 精通 Spring Boot、Spring Cloud
- 熟悉分布式系统设计
- 有高并发经验优先
- 有互联网公司背景

**候选人张三：**
- 学历：硕士，计算机科学 → ✅ 硬性条件通过
- 工作年限：7 年 → ✅
- 技能：Java(精通)、Spring Boot(精通)、Spring Cloud(熟练)、Redis(熟练)、MySQL(精通)、Kafka(熟练)、Docker(熟练)、K8s(了解)
- 项目：某电商平台订单系统重构，日订单量 100w+，从单体架构迁移到微服务
- 公司：前某中型互联网公司
- 简历质量：结构清晰，有量化产出

**打分：**
| 维度 | 分数 | 加权得分 |
|------|------|----------|
| 硬性条件 | ✅ 通过 | 30 |
| 技能栈 | 9 | 22.5 |
| 项目/经验 | 9 | 18 |
| 软性素质 | 7 | 7 |
| 行业/公司 | 8 | 8 |
| 简历质量 | 9 | 4.5 |
| **总分** | | **90** |

**匹配摘要：** 候选人技术栈与 JD 高度吻合，有电商高并发实战经验，项目经历中有系统架构升级经验，很适合该岗位。建议优先安排面试。

### 示例：候选人淘汰

**候选人李四：**
- 学历：大专 → ❌ 淘汰
- 理由：JD 要求"本科及以上"，硬性条件不达标

**处理：** 标记为"硬性条件不匹配"，总分计 0，不进入排名。但如果其他维度特别突出（如 10 年 Java 经验、架构师级别），标记为⭐破格候选，在报告中单独列出供用户参考。
