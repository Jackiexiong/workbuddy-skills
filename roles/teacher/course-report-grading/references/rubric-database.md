# 数据库系统原理 — 课程设计评分专项量规

> 本量规基于《数据库应用课程设计》任务书解读，定义数据库课设的评分标准。
>
> ## 设计架构
>
> - **成绩表分值列**：按任务书三大题给分（50/30/20 = 100），忠实于任务书结构
> - **评价维度**（融入打分逻辑与评语）：任务书的 5 个考察要点
> - **固定 vs 任务相关**：
>   - 三大题分值占比（50/30/20）**固定不变**（每年如此）
>   - 每大题内的具体题目内容**每年微调**（需从当年任务书解读）
>
> ## 任务书原话：评价考察要点（5 条）
>
> 1. **设计正确性**：需求理解、数据库建模正确性、规范化遵循、可扩展性/性能/安全性
> 2. **软件架构与代码质量**（有编程要求时生效）：代码规范性、结构清晰度、容错性/可维护性/可扩展性
> 3. **报告撰写质量**：格式遵循、内容准确性/完整性/逻辑清晰度/文字通顺性
> 4. **优化与创新性**：新架构新技术使用、工程方法创新、综合解决问题能力
> 5. **工作量与难度**：题目难易与工作量大小，适当权重调整以体现公平性

## 三大题评分标准

### 题目 1：数据库建模分析设计题（50 分）

> **固定结构**：需求分析→概念设计(ER图)→逻辑设计(关系模式)→物理设计(建表SQL)
>
> **2025 年题目**：共享科创平台系统需求（企业发布课题、研究团队申请、项目执行监控、结题评价）
>
> ⚠️ 每年业务场景可能更换，但设计流程要求不变。

#### 交付物检查点

- [ ] 需求分析报告（业务需求理解、用户角色、功能模块、非功能需求）
- [ ] ER 设计图（实体、属性、主键、联系类型 1:1/1:N/M:N）
- [ ] 关系模型转换（主码、外码标注）
- [ ] 建表 SQL 脚本（表和字段均加注释）
- [ ] 报告包含上述全部内容

#### 评分参考

| 得分 | 标准 |
|------|------|
| 45–50 | 需求理解准确，ER 图完整规范，关系模式达 3NF+，SQL 全部正确，设计有亮点 |
| 40–44 | 需求理解到位，ER 图基本正确，关系模式规范，SQL 基本正确 |
| 35–39 | 需求理解有偏差，ER 图有遗漏/错误，关系模式存在范式问题 |
| 30–34 | 需求理解不准确，ER 图结构性错误，SQL 多处错误 |
| <30 | 建模设计严重缺失或错误 |

#### 子项评分要点

| 子项 | 评价要点（对应考察要点1：设计正确性） |
|------|------|
| 需求分析 | 是否完整理解业务场景，用户角色/功能模块/非功能需求是否覆盖 |
| ER 概念建模 | 实体/属性/主键/联系类型是否准确，弱实体/ISA 等是否恰当 |
| 关系模式 | ER→关系转换是否正确，主外码标注，规范化程度（至少 3NF） |
| 建表 SQL | DDL 语法正确，类型选择合理，表/字段注释完整 |
| 可扩展性/安全性 | 设计是否考虑扩展，权限/安全是否有设计思考 |

### 题目 2：数据库管理和优化实验（30 分）

> **固定结构**：备份 + 恢复 + 大数据量查询优化
>
> **2025 年题目**：
> - 2.1 Linux 下 MySQL 定时备份
> - 2.2 手动备份 + 误删恢复测试
> - 2.3 百万级数据表 5 个统计查询 + 索引优化 + 执行时间对比

#### 检查点

**2.1 数据库备份（约 8 分）**
- [ ] Linux 系统（CentOS/Ubuntu）环境搭建说明
- [ ] 定时任务（crontab）配置正确
- [ ] 备份脚本可运行，每日自动执行
- [ ] 备份文件命名/存储规范

**2.2 数据库恢复（约 7 分）**
- [ ] 手动备份操作完整记录
- [ ] 误删表记录的模拟操作
- [ ] 根据备份恢复数据的完整过程
- [ ] 恢复结果验证

**2.3 查询优化（约 15 分）**
- [ ] 5 个统计查询全部实现：
  - 每个学生选课门数（学号/姓名/门数）
  - 指定老师课程的所有学生（学号/姓名/班级）
  - 每门课选课人数（课程编号/名称/人数）
  - 每门课成绩分布（课程/不及格/70-80/80-90/≥90 人数）
  - 各班级男女生人数统计
- [ ] 优化前查询耗时截图
- [ ] 索引创建与优化 SQL
- [ ] 优化后耗时对比
- [ ] 优化过程说明（不只是给最终结果）

#### 评分参考

| 得分 | 标准 |
|------|------|
| 27–30 | 备份恢复完整，5 个查询全部正确且优化过程详实，索引设计合理 |
| 24–26 | 备份恢复完整，查询基本正确，有优化但过程不够详尽 |
| 21–23 | 备份或恢复有缺项，查询有错误，优化不充分 |
| 18–20 | 多项缺失，查询错误较多 |
| <18 | 管理优化实验严重不完整 |

---

### 题目 3：数据库应用开发（20 分）

> **固定结构**：存储过程实现数据加工处理（清洗→归档→统计）
>
> **2025 年题目**：订单数据归档与统计（orders_raw → orders_archive + customer_stats）
>
> ⚠️ 业务场景每年可能更换，但存储过程+游标+事务+异常处理的技术要求不变。

#### 检查点

- [ ] 三张表结构正确创建（orders_raw / orders_archive / customer_stats）
- [ ] 存储过程封装完整处理逻辑
- [ ] 数据清洗转换：仅处理 completed 订单，计算金额/折扣/最终金额
- [ ] 客户等级划分逻辑正确（VIP/高级/普通）
- [ ] 数据归档到 orders_archive，标记已处理
- [ ] 月度统计写入 customer_stats（订单数/总金额/平均金额/最常购买地区/等级）
- [ ] 游标遍历原始数据
- [ ] 事务确保数据一致性
- [ ] 异常处理机制
- [ ] 临时表进行中间计算

#### 评分参考

| 得分 | 标准 |
|------|------|
| 18–20 | 存储过程完整正确，游标/事务/异常/临时表全部运用，代码规范清晰 |
| 16–17 | 存储过程基本正确，技术要素大部分运用，小瑕疵 |
| 14–15 | 存储过程有逻辑错误，部分技术要素缺失 |
| 12–13 | 存储过程不完整，多处错误 |
| <12 | 应用开发严重缺失或无法运行 |

#### 子项评分要点

| 子项 | 评价要点（对应考察要点2：软件架构与代码质量） |
|------|------|
| 代码规范性 | SQL 缩进/命名/注释规范 |
| 结构清晰度 | 存储过程逻辑分层清晰，易读 |
| 容错性 | 异常处理是否完善 |
| 可维护性 | 是否模块化，是否便于修改 |
| 技术正确性 | 游标/事务/临时表使用是否正确 |

## 跨题目评价维度（融入评语）

> 以下 3 个考察要点不单独给分，但在每题打分时综合考虑，并在综合评语中体现：

| 考察要点 | 评价方式 |
|---------|---------|
| **报告撰写质量** | 影响每题得分（格式混乱/内容缺失扣分），并在评语中具体指出文档问题 |
| **优化与创新性** | 在题目 2（查询优化）和题目 3（存储过程）中体现，有创新加分 |
| **工作量与难度** | 作为总分微调因素（±3 分内），在评语中说明难度考量 |

---

## 综合评语撰写要求

每篇报告生成 150–200 字综合评语，结构：

1. **总体评价**（1-2 句）：三大题完成情况概述 + 等级定性
2. **题目 1 评价**（2-3 句）：建模设计的优缺点，引用 ER 图/关系模式中的具体问题
3. **题目 2 评价**（2-3 句）：管理优化实验完成度，查询优化的亮点/不足
4. **题目 3 评价**（2-3 句）：存储过程实现质量，代码规范性
5. **改进建议**（1-2 句）：针对该学生的具体提升方向

> 评语应具体、有针对性，引用报告中的具体内容佐证，避免泛泛而谈。

## 每年任务书更新指引

> 当教师提供新一年的任务书时，仅需更新以下部分（三大题分值占比不变）：

1. **题目 1 业务场景**：替换"共享科创平台"为当年题目，但设计流程要求（ER→关系→SQL）不变
2. **题目 2 查询题目**：5 个统计查询的具体内容可能微调，检查点同步更新
3. **题目 3 业务场景**：替换"订单归档"为当年题目，但存储过程+游标+事务+异常处理的技术要求不变

> 任务书的三大题分值（50/30/20）和 5 个考察要点**每年固定**，无需改动。
