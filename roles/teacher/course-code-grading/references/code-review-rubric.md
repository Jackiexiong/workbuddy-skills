# 代码评分量规 — Code Review Rubric

本文档详细定义 5 维度评分标准，供 AI 在评审代码时参考，确保评分的一致性和可解释性。评分按编程语言分类提供细化标准。

---

## 总体评分公式

```
总分 = 功能完整性 × 40%
     + 代码质量 × 25%
     + 技术正确性 × 15%
     + 项目结构与工程化 × 10%
     + 创新与拓展 × 10%
```

每个维度给 0-10 分，按权重加权得总分（满分 100）。

---

## 一、功能完整性（权重 40%）

### 评分标准

| 分数 | 描述 |
|------|------|
| 10 | 全部功能完整实现，包括边缘情况和异常处理 |
| 8-9 | 核心功能全部实现，少数次要功能缺失或有小瑕疵 |
| 6-7 | 核心功能基本实现，有 1-2 项功能未完成 |
| 4-5 | 部分核心功能实现，多项功能缺失 |
| 2-3 | 只有少量功能雏形，大部分未实现 |
| 0-1 | 代码与任务书要求无关或几乎无实现 |

### 通用检查清单

对每个功能需求，检查以下方面：

- [ ] 该功能是否在代码中存在对应的类/函数/模块？
- [ ] 功能入口是否正确（URL 路由、命令行参数、菜单选项等）？
- [ ] 基本业务流程是否跑通？
- [ ] 异常输入是否处理（空值、非法值、越界等）？
- [ ] 边界条件是否覆盖？

### 语言/框架特定的功能检查

| 类型 | 检查项 |
|------|--------|
| Web 项目 | 路由是否映射正确、API 响应格式是否规范、页面是否能正常渲染 |
| 数据库项目 | SQL 是否正确、数据 CRUD 是否完整、事务处理是否到位 |
| GUI 项目 | 界面控件是否对应功能、交互逻辑是否正确 |
| CLI 项目 | 命令行参数解析是否正确、输出格式是否规范 |

## 二、代码质量（权重 25%）

### 评分标准

| 分数 | 描述 |
|------|------|
| 10 | 代码优雅、高度可读、模块化优秀、注释恰到好处 |
| 8-9 | 代码规范清晰、命名合理、有适量注释 |
| 6-7 | 代码基本规范、可读性可接受、部分区域需改进 |
| 4-5 | 代码组织混乱、命名不规范、缺少必要注释 |
| 2-3 | 代码难以阅读、大量重复、命名随意 |
| 0-1 | 代码完全无结构、无注释、不可维护 |

### 细分标准

#### 代码规范性（权重 30%）

按不同语言的命名规范检查：

| 语言 | 类名 | 方法/函数 | 变量 | 常量 |
|------|------|-----------|------|------|
| Java | PascalCase | camelCase | camelCase | UPPER_SNAKE |
| Python | PascalCase | snake_case | snake_case | UPPER_SNAKE |
| JavaScript | PascalCase | camelCase | camelCase / camelCase | UPPER_SNAKE |
| C/C++ | PascalCase | camelCase/snake_case | snake_case | UPPER_SNAKE |

结构规范检查点：
- [ ] 缩进是否统一（空格 vs Tab）
- [ ] 大括号风格是否一致
- [ ] 行长度是否在合理范围（通常不超过 120 字符）
- [ ] 文件/类是否职责单一

#### 注释质量（权重 20%）

| 级别 | 描述 |
|------|------|
| 优秀 | 关键逻辑有注释、API 有文档、复杂算法有解释 |
| 合格 | 主要类和方法有注释、难懂逻辑有说明 |
| 不足 | 几乎无注释、注释与代码不符 |
| 差 | 完全无注释或注释全部是自动生成 |

#### 模块化设计（权重 25%）

- [ ] 类/函数职责是否单一
- [ ] 是否合理分层（如 Controller/Service/DAO）
- [ ] 重复代码是否已提取复用
- [ ] 是否过度设计

#### 健壮性（权重 25%）

- [ ] 是否有异常处理机制
- [ ] 输入参数是否有校验
- [ ] 资源是否正确释放（文件流、数据库连接等）
- [ ] 是否有日志记录

---

## 三、技术正确性（权重 15%）

### 评分标准

| 分数 | 描述 |
|------|------|
| 10 | 技术方案合理、无缺陷、有测试覆盖 |
| 8-9 | 技术方案合理、少量小缺陷（不影响核心功能） |
| 6-7 | 技术方案基本可行、有一些不严重的问题 |
| 4-5 | 技术方案有较大缺陷或使用了不恰当的技术 |
| 2-3 | 技术方案严重错误、代码有严重 Bug |
| 0-1 | 代码完全不可运行或有毁灭性缺陷 |

### 检查清单

#### 通用技术问题

- [ ] 是否存在明显的逻辑错误
- [ ] SQL 注入 / XSS / CSRF 等安全风险
- [ ] 硬编码敏感信息（数据库密码、API Key）
- [ ] 循环/递归是否存在死循环风险
- [ ] 并发/多线程是否存在竞态条件

#### 按语言分类的常见问题

| 语言 | 常见技术问题 |
|------|-------------|
| Java | NullPointerException 风险、未关闭流、未处理受检异常 |
| Python | 类型错误、资源未释放、全局变量滥用 |
| JavaScript | 异步处理不当、回调地狱、this 指向错误 |
| C/C++ | 内存泄漏、缓冲区溢出、野指针 |
| SQL | N+1 查询、缺少索引、SQL 注入 |

---

## 四、项目结构与工程化（权重 10%）

### 评分标准

| 分数 | 描述 |
|------|------|
| 10 | 项目结构专业、构建工具配置完善、有 CI 配置文件 |
| 8-9 | 结构合理、使用标准构建工具、配置文件完整 |
| 6-7 | 基本结构、有构建工具但有瑕疵 |
| 4-5 | 结构混乱或缺少构建工具 |
| 2-3 | 所有文件堆在一起、无构建配置 |
| 0-1 | 无结构可言 |

### 检查清单

#### 目录结构

```
优秀结构示例：
src/
├── main/
│   ├── java/
│   │   └── com/example/
│   │       ├── controller/    ← 控制器层
│   │       ├── service/       ← 业务逻辑层
│   │       ├── dao/           ← 数据访问层
│   │       ├── model/         ← 数据模型
│   │       └── config/        ← 配置类
│   └── resources/
│       ├── application.yml
│       └── db/migration/
├── test/
└── pom.xml / build.gradle
```

| 检查项 | 说明 |
|--------|------|
| 分层结构 | Controller/Service/DAO 或类似分层 |
| 配置文件 | application.yml、.env、requirements.txt 等 |
| 构建工具 | Maven/Gradle/npm/pip 等 |
| 依赖管理 | 有依赖列表（pom.xml/package.json/requirements.txt） |
| .gitignore | 是否排除 IDE 文件、target 目录等 |
| README | 是否有项目说明文档 |

---

## 五、创新与拓展（权重 10%）

### 评分标准

| 分数 | 描述 |
|------|------|
| 10 | 有显著的创新功能或优秀的技术亮点 |
| 8-9 | 有额外功能或性能优化，实现质量高 |
| 6-7 | 有少量额外功能或合理的技术改进 |
| 4-5 | 仅实现基本要求，无额外内容 |
| 2-3 | 连基本要求都未完全覆盖 |
| 0-1 | 严重缺失基本功能 |

### 加分项

以下情况可加分（每项 +1~2 分）：

- [ ] 实现了任务书中标记为"选做"或"拓展"的功能
- [ ] 有单元测试或集成测试
- [ ] 使用了先进的框架或技术（在课程范围之外主动学习）
- [ ] 有性能优化措施（缓存、索引优化等）
- [ ] 有良好的 UI/UX 设计
- [ ] 有部署脚本（Dockerfile、CI 配置等）
- [ ] 有 API 文档

---

## 评分扣分项

以下情况在对应维度中扣分：

| 扣分项 | 所属维度 | 扣分幅度 |
|--------|---------|---------|
| 代码与任务书无关（照抄网络代码未修改） | 功能完整性 | -3~-5 |
| 代码无法编译/运行 | 技术正确性 | -3~-5 |
| 数据库密码等敏感信息硬编码 | 技术正确性 | -2~-4 |
| 明显抄袭（多份学生代码雷同） | 所有维度 | 标记疑似抄袭 |
| 没有 Git 提交历史（一次提交全部内容） | 项目结构 | -1~-2 |
| 包含第三方库的二进制文件 | 项目结构 | -1 |

## 评分示例

### 示例 1：Java Web 项目（优秀）

**项目概况：** Spring Boot 学生管理系统，实现了 CRUD + 分页查询 + Excel 导出

**评分：**

| 维度 | 分数 | 依据 |
|------|------|------|
| 功能完整性 | 9 | 全部功能实现，完美运行，额外做了导出功能 |
| 代码质量 | 8 | 命名规范、分层清晰，但部分方法偏长 |
| 技术正确性 | 9 | 方案合理、有异常处理、少量日志不完整 |
| 项目结构 | 10 | Maven 项目、标准三层架构、有 .gitignore |
| 创新与拓展 | 8 | 实现了 Excel 导出、使用了缓存 |

总分 = 9×4 + 8×2.5 + 9×1.5 + 10×1 + 8×1 = 36+20+13.5+10+8 = **87.5 ≈ 88 分（良好）**

### 示例 2：Python 数据分析项目（不及格）

**项目概况：** 简单的数据读取 + 一个没写完的统计分析

**评分：**

| 维度 | 分数 | 依据 |
|------|------|------|
| 功能完整性 | 3 | 只实现了数据读取，核心分析功能未完成 |
| 代码质量 | 5 | 命名尚可，但无注释、无函数拆分 |
| 技术正确性 | 4 | 读取部分正确，但有潜在编码问题 |
| 项目结构 | 2 | 单文件、无 requirements.txt |
| 创新与拓展 | 0 | 无任何额外内容 |

总分 = 3×4 + 5×2.5 + 4×1.5 + 2×1 + 0×1 = 12+12.5+6+2+0 = **32.5 ≈ 33 分（不及格）**

## 常见评分偏误与校准

| 偏误类型 | 说明 | 校准方法 |
|---------|------|---------|
| 代码量偏误 | 代码多≠质量高 | 重点检查功能完整性和代码质量，而非行数 |
| 熟悉度偏误 | 熟悉的语言容易给高分 | 所有维度都有客观检查清单 |
| 首因效应 | 第一个项目影响后续评分 | 评 5 个后回顾前几个，调整一致性 |
| 晕轮效应 | 一个维度好导致其他维度评分偏高 | 逐维度评分，先评完所有学生的同一维度 |
